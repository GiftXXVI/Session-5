Essential Knowledge üìö

APIs ‚öô

What is an API?

  • API stands for Application Programming Interface.
  • APIs form the foundation of IOT, Mobile Apps and Cloud.
  • APIs provide a simple interface (set of available methods) that allow other software applications to perform complex tasks.

Why do we need APIs?

  • They provide a simple interface for other applications.
  • They make it easy for developers to use advanced and complex functionality.

Examples:

  • NVidia CUDA allows Python applications to run Python code on GPUs(for AI, Blockchain ...).
  • Psycopg2 allows Python applications to run SQL queries on PostGreSQL.
  • Web pages can get the GPS location of a device from the browser via a Geolocation API.

REST APIs ‚öô

REST APIs

  • An architectural style and guideline that applies to web based APIs.
  • Roy Fielding defined REST in a PHd dissertation in 2000.
  • APIs that fulifill the REST constraints are called RESTful APIs.

REST APIs(Continued)

  • REST APIs are typically based on HTTP Protocol and HTTP Methods such as GET, POST, DELETE.
  • Their output can be JSON, XML, HTML or BLOB (Binary Large Object).
    • The typical format in modern REST APIs is JSON.
    •

    REST APIs(Continued)

    • Creating a REST API creating an API involves abstracting away the underlying implementation details (DBMS, Database Structure ...) and presenting an easy to use interface to the client application.
    • A REST API is fairly small server-side application, sometimes implemented as a micro-service.
    • Clients access resources using URIs and send and receive data using a specified format.

    Architectural Constraints

    • Client-Server Architecture: REST Architecture enforces a Client-server architecture thereby enforcing separation of concerns between user interface(frontend) and backend.
    • Statelessness: REST Architecture is stateless (it keeps no session information, each request is processed in isolation), thereby eliminating the overhead of maitaining sessions. This also means you can load balance requests between many servers.

    Architectural Constraints

    • Cacheability: As a result of the Stateless nature of the REST Architecture Responses can be Cached; another property that supports scalability.
    • Layered: The application architecture is supposed to be layered such that Caching can be done at multiple layers between the server and the client.

    Architectural Constraints

    • Uniform interface:
      • Individual Objects should be identifiable via a URI.
      • If a client holds the information of an Object, it should be able to modify or delete the Object.
      • Responses must include enough information to process the Response (for example: Content-Type Header).
      • *** If a client accesses the base URL, the client should be able to discover all available objects that it needs.

    Architectural Constraints

    • *** Code on demand: Client should be able to download some executable code from the server to extend functionality.

    Why do we need REST APIs?

    • They allow us to develop scalable APIs from micro-services.
    • This ties in to our goal to create applications that will scale.

    HTTP üåå

    Methods üì®

    GET

    Used to retrieve an item (with an id/URI) or items (with a URL).

    POST

    Used to create an item.

    PUT

    Used to modify by replacing/overwriting all properties of an item.

    PATCH

    Used to modify by changing a subset of properties.

    OPTIONS

    Used to specify allowed methods (especially for crossorigin requests).

    Status Codes üö¶

    Success Responses

    Code Description
    200 The request succeeded.
    201 The request succeeded, and a new resource was created as a result.
    202 The request has been received but not yet acted upon. In this case, there will never be a final result code.

    Redirection Responses

    Code Description
    301 The URL of the requested resource has been changed permanently. The new URL is given in the response.
    304 This is used for caching purposes. It tells the client that the resource has not been modified thus it can use a cached version.

    Client Error Responses

    Code Description
    400 The server detected a malformed request syntax, invalid request message framing...
    401 The client must authenticate itself first to get the requested response.
    403 The client does not have access rights to the content.
    404 The server can not find the requested resource.

    Server Error Responses

    Code Description
    500 The server has encountered a situation it does not know how to handle.
    503 The server is not ready to handle the request. Common causes are a server that is down for maintenance or that is overloaded.
